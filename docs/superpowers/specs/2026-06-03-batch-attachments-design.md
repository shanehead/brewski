# Batch Attachments Design

**Date:** 2026-06-03
**Status:** Approved

## Problem

Batches have no way to attach brew-day photos, fermentation shots, water reports, hop invoices, or any other files. Homebrewers want a record of the full brew alongside the numbers.

## Solution Overview

A dedicated **Attachments tab** in the batch view (alongside Overview · Gravity · Tasting) that holds any mix of photos and files. Images render as a masonry photo wall; non-image files appear as compact inline rows beneath. Files are copied into `{appDataDir}/attachments/{batch_id}/` at attach time — photos are resized and JPEG-encoded, other files are copied raw.

## Architecture

### Approach chosen

Copy files into app data dir; transcode images via existing `write_image`. This keeps attachments portable (travel with the appDataDir backup), bounds photo storage, and is consistent with the recipe image pattern.

## Section 1: Data & Storage

### Migration

**`src-tauri/migrations/011_batch_attachments.sql`:**

```sql
CREATE TABLE batch_attachments (
    id            TEXT    PRIMARY KEY,
    batch_id      TEXT    NOT NULL REFERENCES batches(id) ON DELETE CASCADE,
    filename      TEXT    NOT NULL,
    original_name TEXT    NOT NULL,
    mime_type     TEXT,
    size_bytes    INTEGER NOT NULL,
    created_at    INTEGER NOT NULL
);
```

- `filename` — UUID-based on-disk name, e.g. `"a1b2c3d4.jpg"`
- `original_name` — user-facing display name, e.g. `"brew-day.jpg"`
- `mime_type` — detected by Rust from file extension; `"image/jpeg"`, `"application/pdf"`, etc.
- `ON DELETE CASCADE` — DB row removed when the parent batch is deleted; Rust delete command also removes the file

### File storage layout

```
{appDataDir}/
  attachments/
    {batch_id}/
      {uuid}.jpg       ← transcoded photos
      {uuid}.pdf       ← raw non-image files
      {uuid}.docx
```

**Image detection:** file extension in `[jpg, jpeg, png, gif, webp, heic]` → treated as image, resized to max 1200 px and JPEG-encoded at 85% quality via the existing `write_image` helper. All other extensions copied raw via `fs::copy`.

## Section 2: Rust Backend

### New files

| File | Responsibility |
|------|----------------|
| `src-tauri/src/commands/batch_attachments.rs` | 4 Tauri commands |
| `src-tauri/src/repositories/batch_attachments.rs` | SeaORM DB access |
| `src-tauri/src/entities/batch_attachments.rs` | Generated by `sea-orm-cli` |

### Storage helper

`attachments_dir(app, batch_id)` → `{appDataDir}/attachments/{batch_id}/`

Mirrors the existing `images_dir(app)` pattern in `recipe_image.rs`.

### Commands

**`add_batch_attachment(batch_id, source_path, original_name)`**
1. Detect file extension; determine `mime_type` string and whether to transcode
2. Generate UUID filename with appropriate extension
3. Call `write_image(src, dest)` for images, `fs::copy` for everything else
4. Record `size_bytes` from the written file
5. Insert `batch_attachments` row, return `BatchAttachment`

**`list_batch_attachments(batch_id)`** → `Vec<BatchAttachment>`
Ordered by `created_at ASC`.

**`delete_batch_attachment(id)`**
1. Look up the row; compute full path via `attachments_dir`
2. Delete file (ignore `NotFound`)
3. Delete DB row

**`open_batch_attachment(id)`**
1. Look up row; compute full path
2. Call `tauri_plugin_opener::open_path(path, None)`

### `BatchAttachment` model

Added to OpenAPI spec, regenerated into `api.gen.ts` and `models.gen.rs`:

```

batch_id      string

original_name string
mime_type     string | null

```

## Section 3: Frontend

### New component

**`src/lib/components/batch/BatchAttachmentsTab.svelte`**

Props: `{ batch: Batch }`

Layout (photo wall + inline files):
- On mount: call `list_batch_attachments(batch.id)`, split results into `photos` (mime starts with `image/`) and `files`
- **Photo wall:** CSS grid, 3 columns desktop / 2 columns mobile. Each cell defaults to `aspect-ratio: 1` with `object-fit: cover` (square thumbnail). If the image's natural height exceeds 1.3× its natural width (portrait), an `onload` handler on the `<img>` sets `grid-row: span 2` on the parent cell. Each cell loads via `convertFileSrc(appDataDir + "/attachments/" + batch.id + "/" + attachment.filename)`. Tap → `open_batch_attachment(id)`. Delete (×) button on each cell → `delete_batch_attachment(id)`, then refresh list
- **Inline file rows:** icon derived from extension (📄 default, 📊 spreadsheet, 🖼️ if image/*, etc.), `original_name`, human-readable `size_bytes`. Tap row → `open_batch_attachment(id)`. Delete (×) → `delete_batch_attachment(id)`, then refresh list
- **"+ Add" button** at top of tab → `dialog.open({ multiple: true })` → for each returned path call `add_batch_attachment(batch.id, path, basename(path))`, then refresh list

### Platform updates

- **`src/lib/desktop/BatchView.svelte`** — add `{ key: "attachments", label: "Attachments" }` to `TABS`, render `<BatchAttachmentsTab {batch} />` in the `{#if activeTab === "attachments"}` branch
- **`src/lib/mobile/BatchView.svelte`** — same tab addition; `BatchAttachmentsTab` is shared (lives in `components/batch/`, not `desktop/` or `mobile/`)

### Mobile specifics

- Same `BatchAttachmentsTab.svelte` used on both platforms
- `dialog.open()` routes to native file picker on iOS/Android
- Masonry grid: 3 columns → 2 columns at `max-width: 640px`
- Delete buttons: minimum 44px tap target
