# SeaORM Migration Design

## Goal

Replace the `db/` module (direct sqlx queries) with SeaORM + the repository pattern. Migrate everything at once. Use `TryFrom` to translate SeaORM entity models into API models.

## Architecture

Three new modules replace `db/` entirely:

- `migration/` — SeaORM migrations, run at startup
- `entities/` — generated by `sea-orm-cli`, committed to repo, schema-first
- `repositories/` — one repository struct per domain

`commands/`, `models.rs`, `brewing/`, and `error.rs` are unchanged or lightly modified.

## Dependencies

Remove `sqlx`. Add to `src-tauri/Cargo.toml`:

```toml
[dependencies]
sea-orm = { version = "1", features = ["sqlx-sqlite", "runtime-tokio-native-tls", "macros", "with-uuid"] }
sea-orm-migration = { version = "1", features = ["sqlx-sqlite", "runtime-tokio-native-tls"] }
```

`sea-orm-cli` is installed as a dev tool (`cargo install sea-orm-cli`), not in Cargo.toml.

## Migration Module

`migration/mod.rs`:

```rust
use sea_orm_migration::prelude::*;

mod m001_initial;
mod m002_seed_data;

pub struct Migrator;

#[async_trait::async_trait]
impl MigratorTrait for Migrator {
    fn migrations() -> Vec<Box<dyn MigrationTrait>> {
        vec![
            Box::new(m001_initial::Migration),
            Box::new(m002_seed_data::Migration),
        ]
    }
}
```

`m001_initial.rs` translates the current `001_initial.sql` into SeaORM `Table::create()` DDL statements — one per table, all in `up()`, all dropped in `down()`.

`m002_seed_data.rs` runs the existing SQL file as a raw statement:

```rust
manager.get_connection()
    .execute_unprepared(include_str!("../db/migrations/002_seed_data.sql"))
    .await?;
```

The raw SQL files stay on disk until the migration module is verified working, then `db/migrations/` is removed.

## AppState & Startup

```rust
use sea_orm::{Database, DatabaseConnection};
use sea_orm_migration::MigratorTrait;
use crate::migration::Migrator;

pub struct AppState {
    pub db: DatabaseConnection,
}

// in setup():
let db_path = app_dir.join("brewski.db");
let db = Database::connect(format!("sqlite://{}?mode=rwc", db_path.display())).await?;
Migrator::up(&db, None).await?;
app.manage(AppState { db });
```

## Entities

Generated by `sea-orm-cli` with `--with-serde both --serde-skip-hidden-column`. Each entity exports `Entity`, `Model`, `Column`, `ActiveModel`, and `Relation`.

The `type` reserved-word column is generated as `r#type` in the model struct. The `TryFrom` impls handle the rename to `type_`.

### Regeneration workflow (encoded in Justfile)

`sea-orm-cli migrate up` requires a standalone binary, so a small helper is added:

**New file:** `src-tauri/src/bin/migrate.rs`

```rust
#[tokio::main]
async fn main() {
    let url = std::env::args().nth(1).expect("usage: migrate <sqlite-url>");
    let db = sea_orm::Database::connect(&url).await.unwrap();
    brewski_lib::migration::Migrator::up(&db, None).await.unwrap();
    println!("Migrations applied.");
}
```

Justfile recipes:

```makefile
# Apply SeaORM migrations to a local dev database
migrate:
    cargo run --manifest-path src-tauri/Cargo.toml --bin migrate -- sqlite://./dev.db

# Regenerate SeaORM entities from the dev database (run `just migrate` first)
gen-entities: migrate
    sea-orm-cli generate entity \
      --database-url sqlite://./dev.db \
      --output-dir src-tauri/src/entities \
      --with-serde both \
      --serde-skip-hidden-column
```

`just gen-entities` depends on `just migrate` so the dev DB is always up to date before entity generation.

## TryFrom Impls

All `TryFrom` impls live in `models.rs`, directly below their API struct. Error type is `AppError`.

**Flat conversions** (one entity model → one API model) for:
`RecipeAdditionFermentable`, `RecipeAdditionHop`, `RecipeAdditionYeast`, `RecipeAdditionMisc`, `RecipeAdditionWater`, `MashStep`, `EquipmentProfile`, `Style`, `Fermentable`, `Hop`, `Yeast`, `Misc`, `Water`

Example:

```rust
impl TryFrom<entities::recipe_addition_fermentables::Model> for RecipeAdditionFermentable {
    type Error = AppError;
    fn try_from(m: entities::recipe_addition_fermentables::Model) -> Result<Self, AppError> {
        Ok(Self {
            id: m.id,
            recipe_id: m.recipe_id,
            fermentable_id: m.fermentable_id,
            name: m.name,
            type_: m.r#type,
            yield_pct: m.yield_pct,
            color_lovibond: m.color_lovibond,
            amount_kg: m.amount_kg,
            add_after_boil: m.add_after_boil,
            addition_order: m.addition_order,
        })
    }
}
```

**Nested assembly** — `Recipe` and `Mash` embed child collections and cannot be produced by a single `TryFrom`. Repositories assemble them using `TryFrom` on each component:

```rust
let fermentables = recipe_addition_fermentables::Entity::find()
    .filter(recipe_addition_fermentables::Column::RecipeId.eq(id))
    .order_by_asc(recipe_addition_fermentables::Column::AdditionOrder)
    .all(self.db).await?
    .into_iter()
    .map(RecipeAdditionFermentable::try_from)
    .collect::<Result<Vec<_>, _>>()?;
```

**`RecipeSummary`** needs the joined style name. SeaORM's `find_also_related` returns `Vec<(recipe::Model, Option<style::Model>)>`. The repository maps this tuple directly — no `TryFrom` for this one since `TryFrom` takes a single input.

**Removed from `models.rs`:** `RecipeRow` and `MashRow` — SeaORM entity models serve this role directly.

## Repositories

Each repository holds a reference to `DatabaseConnection`:

```rust
pub struct RecipeRepository<'a> {
    db: &'a DatabaseConnection,
}

impl<'a> RecipeRepository<'a> {
    pub fn new(db: &'a DatabaseConnection) -> Self { Self { db } }
}
```

| File | Struct | Methods |
|---|---|---|
| `recipe.rs` | `RecipeRepository` | list, get, create, update, delete |
| `addition.rs` | `AdditionRepository` | create/update/delete × 5 ingredient types |
| `library.rs` | `LibraryRepository` | list_styles, list_fermentables, list_hops, list_yeasts, list_miscs, list_waters |
| `equipment.rs` | `EquipmentRepository` | list, create, update, delete |
| `mash.rs` | `MashRepository` | get, upsert, create_step, update_step, delete_step, update_step_order |
| `settings.rs` | `SettingsRepository` | get_all, update |

## Commands

Commands become one-liners:

```rust
#[tauri::command]
pub async fn list_recipes(state: State<'_, AppState>) -> Result<Vec<RecipeSummary>, String> {
    RecipeRepository::new(&state.db).list().await.map_err(|e| e.to_string())
}
```

## Testing

`test_helpers::setup_test_db()` returns a `DatabaseConnection` (was `SqlitePool`), connects to an in-memory SQLite DB, and runs `Migrator::up()`. All existing tests call repository methods directly — test structure is unchanged.

```rust
pub async fn setup_test_db() -> DatabaseConnection {
    let db = Database::connect("sqlite::memory:").await.unwrap();
    Migrator::up(&db, None).await.unwrap();
    db
}
```
